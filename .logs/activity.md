That live-stack pass caught 3 real bugs no test had caught:
1. **docker-compose.yml was never updated for Sprint 2's new env vars** ŌĆö `ai-analysis-service` and `contracts-service` both crash-looped on startup (`Could not resolve placeholder 'INTERNAL_SERVICE_TOKEN'`) because the compose file's `environment:` blocks were never touched when those variables were introduced. Fixed: added `INTERNAL_SERVICE_TOKEN`, `AI_ANALYSIS_SERVICE_URL`, `R2_ENDPOINT_URL`, `CONTRACT_MAX_FILE_SIZE_MB` to both service blocks.
2. **No local S3-compatible service for R2** ŌĆö contracts-service needs R2 to do anything, and `.env` only had placeholder Cloudflare credentials (flagged as a known gap at Sprint 2's end, but not actually addressed until it blocked this verification pass). Added a `localstack` service to docker-compose.yml (mirroring what the test suite already uses) with an init script (`docker/localstack-init/01-create-bucket.sh`) that auto-creates the bucket; `.env` now points R2_* at it for local dev, `.env.example` documents both paths.
3. **A real, cross-cutting data-loss bug**: every entity in the codebase (`User`, `RefreshToken`, `UserProfile`, `Contract`, `AnalysisResult`, `ClauseFlag`) assigns its own UUID id in its constructor rather than using `@GeneratedValue`. Spring Data's default `isNew()` check just tests whether the id is null ŌĆö since these ids are never null, every `save()` call was silently routed through `entityManager.merge()` instead of `persist()`. `merge()` returns a *different* managed instance than the one passed in, so any mutation made to the original in-memory object *after* the initial save (e.g. `contract.setStatus(FAILED)`) was completely lost ŌĆö never reached the entity Hibernate actually flushed. Concretely: a contract that failed AI analysis showed `"status":"FAILED"` in the immediate HTTP response (read from the stale, never-flushed original object) but was verified via direct DB query to be stuck at `PENDING` forever. `createdAt` also came back `null` in the same response for the identical reason, despite being correctly stored in the DB the whole time.
   Fixed at the root: added `ma.mizan.common.persistence.UuidEntity`, a `@MappedSuperclass` implementing Spring Data's `Persistable<UUID>` with an explicit transient `isNew` flag flipped by `@PostPersist`/`@PostLoad` ŌĆö the standard, documented Spring Data JPA pattern for app-assigned identifiers. Retrofitted all 6 entities across all 3 services to extend it instead of declaring their own `@Id` field. `ContractService.upload()` now captures `saveAndFlush()`'s return value and relies on correct dirty-checking (removed a since-unnecessary explicit `save()` that had been masking symptoms of this same bug during Story 3.2 development without me diagnosing the actual cause at the time).
   Re-verified live after the fix: uploaded the same test PDF again ŌĆö response and DB now agree (`FAILED` in both, `createdAt` populated immediately). Full reactor `./mvnw verify` still green (added a unit test for the new `UuidEntity` to keep mizan-common's coverage gate passing).
Not yet re-pushed ŌĆö this fix landed mid-story-verification; will push once Story 4.1/4.2 or a natural sprint checkpoint is reached, unless the fix should go out sooner given its severity (worth discussing with the user).
